# Community & Open Source Strategy

## When to use
- Preparing for public repository launch
- Setting up community infrastructure (Discord, forums)
- Defining contribution guidelines
- Planning community events or content

## Steps

1. **Prepare repos for public**:
   - [ ] README.md — clear, professional, with badges
   - [ ] CONTRIBUTING.md — how to contribute
   - [ ] CODE_OF_CONDUCT.md — community standards
   - [ ] LICENSE — AGPL-3.0
   - [ ] CLA.md — Contributor License Agreement
   - [ ] Issue templates — bug report, feature request
   - [ ] PR template — description, testing, screenshots
   - [ ] .github/FUNDING.yml — sponsorship links

2. **Set up community channels**:
   - GitHub Discussions — primary async communication
   - Discord server — real-time chat
   - Twitter/X @AsgardAIDev — announcements
   - Blog at asgardai.dev — tutorials, releases, case studies

3. **Define governance model**:
   ```
   Governance: Benevolent Dictator For Life (BDFL)
   - BDFL: Project founder (final decision authority)
   - Core team: Commit access, review PRs
   - Contributors: PRs, issues, docs
   - Community: Users, feedback, bug reports
   ```

4. **Plan content calendar** (monthly):
   - Week 1: Release notes / changelog
   - Week 2: Technical deep-dive blog post
   - Week 3: Community spotlight / showcase
   - Week 4: Tutorial / how-to guide

5. **Track community health metrics**:
   - GitHub stars (growth rate)
   - Docker Hub pulls
   - Number of contributors
   - Issue response time (target: < 48h)
   - PR merge time (target: < 1 week)
   - Discord active members

6. **Write community strategy** in `docs/business/community-strategy.md`

## Key principles
- "Community first, product second" — healthy community = sustainable project
- Respond to every issue and PR — even if just to acknowledge
- Make first contribution easy — "good first issue" labels
- Celebrate contributors — shoutouts, credits, CONTRIBUTORS.md
- Be transparent about roadmap and decisions
